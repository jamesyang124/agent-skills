# API Spec Format

This is the canonical format for `docs/agents/api-spec.md` generated by the `sync-api-spec` skill.

---

## File Header

The standard header is always written first:

```markdown
# API Spec — {Project Name}
> Generated by sync-api-spec. Last updated: {YYYY-MM-DD}

## Endpoints Summary

| Method | Path | Purpose | Status |
|--------|------|---------|--------|
| GET    | /api/v1/users/me | Get current authenticated user profile | ✅ |
| POST   | /api/v1/auth/login | Authenticate user, return session token | ✅ |
| POST   | /api/v1/orders | Create a new order | 🔷 CONTRACT |
```

The `Status` column rules:
- `✅` — implemented and verified
- `🔷 CONTRACT` — planned, not yet in router

**Ordering rule**: `🔷 CONTRACT` rows are always listed **first** in the table (above all `✅` rows). Within the CONTRACT rows, the most recently added entry appears first (last-in, stays at top).

When the spec contains any `🔷 CONTRACT` entries, insert the Contract Banner **immediately after the `## Endpoints Summary` table**. Never place it above `# API Spec`:

```markdown
> [!CAUTION]
> **CONTRACT SPEC — Contains unimplemented endpoints**
> Endpoints marked 🔷 CONTRACT in the table above have not been built and are subject to change.
> Remove each badge and this banner once all endpoints are implemented and verified.
```

Immediately after the Contract Banner (or after the Summary table when there is no banner), write the `## Endpoints` section heading. All endpoint cards live inside this section:

```markdown
## Endpoints
```

**Full file layout** (always follow this order exactly):

```
# API Spec — {Project Name}
> Generated by sync-api-spec. Last updated: {YYYY-MM-DD}

## Endpoints Summary
| Method | Path | Purpose | Status |
...

> [!CAUTION] Contract Banner   ← only present when CONTRACT entries exist

## Endpoints

---
## NEWEST CONTRACT endpoint   ← first item in ## Endpoints, most recently added
---
## OLDER CONTRACT endpoint
---
## implemented endpoint        ← implemented cards below all CONTRACT cards
---
## implemented endpoint
```

**Insertion rules**:
- New CONTRACT card → insert as the **first item inside `## Endpoints`** (directly after the `## Endpoints` heading line), before all other cards.
- New implemented card → append after the last implemented card (below all CONTRACT cards).

---

## Per-Endpoint Entry

Each entry follows this structure (one `---` separator between entries):

```markdown
---

## {METHOD} {/path}  🔷 CONTRACT

<!-- ↑ Remove the 🔷 CONTRACT badge once this endpoint is implemented and verified -->

**Purpose**: One-sentence description of what this endpoint does.

**Auth**: `none` | `JWT` | `admin JWT` | `{middleware name}`

**Request**

- Headers:
  - `Authorization: Bearer <token>` (required)
  - `Content-Type: application/json` (required for body endpoints)
  - `X-Request-ID: string` (optional)
- Path params: `id` (string, required) — the resource identifier
- Query params: `page` (int, optional, default: 1)
- Body (`application/json`):
  ```json
  {
    "email": "string",
    "password": "string"
  }
  ```

  | Field | Type | Required | Description |
  |-------|------|----------|-------------|
  | `email` | string | ✓ | User email address |
  | `password` | string | ✓ | Plaintext password (min 8 chars) |

**Response**

| Status | Description | Body shape |
|--------|-------------|------------|
| 200 | Success | `{ token: string, expires_at: string }` |
| 400 | Validation error | `{ error: string }` |
| 401 | Invalid credentials | `{ error: string }` |

**Example (200)**
```json
{
  "token": "eyJhbGci...",
  "expires_at": "2026-05-01T00:00:00Z"
}
```

> **Nested body example** — if the request body has nested fields, list every leaf in the param table and use dot-notation:
>
> Body:
> ```json
> {
>   "user": {
>     "name": "Jane",
>     "address": {
>       "street": "123 Main St",
>       "city": "Taipei"
>     }
>   },
>   "items": [
>     { "productId": "abc", "qty": 2 }
>   ]
> }
> ```
>
> Param table:
>
> | Field | Type | Required | Description |
> |-------|------|----------|-------------|
> | `user.name` | string | ✓ | Display name |
> | `user.address.street` | string | ✓ | Street line |
> | `user.address.city` | string | ✓ | City name |
> | `items[].productId` | string | ✓ | Product SKU |
> | `items[].qty` | int | ✓ | Quantity (min: 1) |

**Implementation notes** *(optional — include only non-obvious details)*
- Invalidates any existing sessions for the user before issuing a new token.
- Calls `AuthService.Authenticate` → `UserRepository.FindByEmail`.
```

## Rules

1. **Incremental**: Do not overwrite existing entries. Only append new ones, update the summary table.
2. **Removed routes**: Mark with a callout, do not delete automatically:
   ```markdown
   > ⚠️ Route removed from router — confirm deletion: DELETE /api/v1/sessions/{id}
   ```
3. **Timestamp**: Always update the `Last updated` date in the file header.
4. **Summary table**: Keep it sorted by path, then method. All documented routes must appear here.
5. **Headers are mandatory**: Every endpoint entry MUST include a `Headers` subsection under Request, even if only `Content-Type` or `none` applies. Never omit it.
6. **Exhaustive param listing**: ALL input fields — required and optional — MUST be listed in the param table, no matter how deeply nested. For nested structures, use dot-notation to show the full path:
   - `user.address.street` (string, optional)
   - `items[].productId` (string, required)
   - `metadata.tags[]` (string[], optional)
   No field may be omitted or summarised as `...` or `// other fields`. If a structure is recursive or unbounded, note the pattern once and give a concrete example.
7. **JSON example completeness**: The example JSON body MUST include all fields listed in the param table — both required and optional — so readers can copy it directly without guessing field names.
8. **Contract badge**: In contract mode, every planned endpoint heading MUST include the `🔷 CONTRACT` badge. Remove it (and offer to remove the banner) only after the implementation is confirmed. Never strip badges automatically during a normal sync.
5. **Implementation notes**: Only include if genuinely useful context — skip for trivial CRUD.
6. **JSON examples**: Always use code blocks. Required for all 2xx responses. Optional for 4xx/5xx if the shape is in the table.
